# UI 設計規範

這份規範讓後續畫面修改維持一致，不靠每次憑感覺調整。

## 1. 資訊順序

1. 第一眼先看標的、價格日期、來源、備援狀態與缺漏。
2. 「已確認資料」與「模型推估」必須分區，不能把估計值寫成已發生事實。
3. 成長性與財務安全永遠分開顯示，不得平均成單一總評級。
4. ETF、一般公司與金融業使用不同文案；不適用時直接顯示原因。
5. 重要限制要放在數字附近，不能只藏在頁尾免責聲明。

## 2. 色彩

色彩來源以 static/css/app.css 的 CSS variables 為準：

| 用途 | Token | 色碼 |
|---|---|---|
| 主文字／深色背景 | --navy-900 | #123047 |
| 主要操作 | --blue-700 | #1d5f8a |
| 成功／低風險 | --green-700 | #167044 |
| 注意／不確定 | --amber-700 | #9a5c08 |
| 錯誤／高風險 | --red-700 | #b4232c |
| 次要文字 | --slate-500 | #596672 |

- 顏色不能是唯一訊息；必須同時有文字、圖示或狀態名稱。
- 一般小字在白底至少使用 --slate-500，不可再降低對比。
- A～F 可以用顏色輔助，但畫面與匯出內容都必須保留字母。

## 3. 字體與字級

- Web：系統 sans-serif，優先 Segoe UI、Microsoft JhengHei、Noto Sans TC。
- PDF／圖表：隨 App 散布的 Noto Sans TC（SIL OFL 1.1）。
- 內文建議 13～16 px；說明文字至少 11 px，長段落至少 12 px。
- 數字使用 tabular numerals，避免更新時跳動。
- 不用全大寫英文取代主要中文標題；英文 eyebrow 只當輔助層級。

## 4. 間距與版面

- 基礎間距：4、8、12、16、24、32 px。
- 卡片內距：桌面 22～28 px，手機 18 px。
- 圓角：主要卡片 12 px，小元件 6～9 px。
- 桌面最大寬度 1440 px；1120、820、560 px 為主要斷點。
- 表格在窄畫面可水平捲動，容器必須可用鍵盤聚焦並有說明。

## 5. 元件狀態

每個會載入資料的元件都要能表達：

- loading：正在取得。
- ready：資料可用。
- missing：資料不足，不能顯示 0 代替。
- stale：使用最後有效舊資料。
- fallback：官方資料不可用，改用已標示的備援。
- not applicable：模型不適用。
- error：來源或格式錯誤，附白話處理方式。

按鈕需有 default、hover、focus-visible、disabled；背景工作另有 cancelling。

## 6. 無障礙

- 所有操作能只用鍵盤完成。
- focus-visible 必須清楚；開啟結果後焦點移到結果區。
- 搜尋建議使用 combobox／listbox／option 語意與 aria-activedescendant。
- 進度與錯誤使用 aria-live，但避免整頁每次更新都重讀。
- 圖表 alt 說明標的與圖表類型；裝飾圖片使用空 alt。
- 點擊目標至少 40×40 px；手機主要按鈕使用整列寬度。

## 7. 文案

- 「參考價」不寫成即時價，除非來源真的提供即時時間。
- 「估值情境區間」不寫成目標價或保證合理價。
- 未通過門檻的模型寫「實驗估計／未通過驗證」。
- 安全篩檢不寫成破產機率，也不代表股價不會下跌。
- 新聞情緒固定標示「關鍵字分類，不是投資訊號」。

## 8. 驗收

每次改 UI 至少檢查：

1. 1440×900、820×1180、390×844。
2. 鍵盤搜尋、取消、閱讀、產生 PDF、下載。
3. 無水平整頁溢出；表格僅在自己的容器內捲動。
4. axe 沒有 critical／serious 問題。
5. JavaScript 語法、Flask 測試與真實資料 smoke test 通過。
